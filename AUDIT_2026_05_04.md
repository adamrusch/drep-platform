### P1 — Significant correctness / UX defects

#### F-01 — DRep delegator count silently capped at 1000

- **Area:** data integrity
- **Evidence:** `backend/src/lib/koios.ts:874-894` calls
  `POST /drep_delegators` without pagination. PostgREST default page size
  is 1000; `Content-Range: 0-999/*` header confirms this on a live probe.
  Yoroi DRep returns exactly 1000 rows; chain has ~10k delegators per
  popular DRep. The detail handler at `backend/src/handlers/directory/get.ts:101-103`
  uses `delegatorsRaw.length` directly as `delegatorCountLive`, so every
  popular DRep displays "1,000 delegators" indefinitely.
- **Fix:** Page the request with `Range: 0-999`, `Range: 1000-1999`, …
  until a short page comes back. Wrap into a helper. Use the same
  pattern Koios already uses for `pool_list` (`MAX_PAGES = 25`).
- **Effort:** S
- **Status:** fixed (see commit)

#### F-02 — Stale data-source labels

- **Area:** docs / DX visible to end users
- **Evidence:**
  - `frontend/src/components/Layout.tsx:253` — sidebar reads
    `"Synced from Blockfrost"`. Per Phase B (commits `118ea5a6`,
    `5de6e114`), Koios is primary; Blockfrost is fallback-only.
  - `frontend/src/pages/DRepDirectoryPage.tsx:484` — page subtitle says
    `"Synced from Koios every five minutes"` but actual cadence is **30
    minutes** (SchedulerStack:170; bumped from 5 → 30 in commit `f6acb024`).
- **Fix:** Update both strings to reflect reality.
- **Effort:** S
- **Status:** fixed (see commit)

#### F-03 — Mesh bundle (1.32 MB gz) preloaded on every page

- **Area:** performance
- **Evidence:** `frontend/dist/index.html` `<link rel="modulepreload" href="/assets/mesh-CfFgn0hx.js">`.
  Raw 6.77 MB, gz 1.32 MB. The chunk plus the `sidan_csl_rs_bg.wasm`
  (5.43 MB) load on every cold cache hit, even on `/governance` and
  `/dreps` where the user isn't connecting a wallet.
  `vite.config.ts:rollupOptions.output.manualChunks` declares `mesh:
  ['@meshsdk/react','@meshsdk/core']` which IS the right move — but
  `WalletAuthProvider` synchronously imports `MeshProvider`, anchoring
  the chunk into the entry graph so Vite emits it as a modulepreload.
- **Fix:** Lazy-load `WalletAuthProvider` (which is what actually pulls
  Mesh). Mount it via `React.lazy(() => import('./auth/WalletAuthProvider'))`
  with a `<Suspense fallback={...children}>` so first-paint pages render
  without waiting for Mesh.
  - Net win: `/governance`, `/dreps`, `/`, `/governance/:id` and every
    other non-wallet route stops paying the 1.32 MB transfer + parse +
    WASM compile cost on first visit.
  - The Mesh module loads when the auth provider mounts (i.e. when a
    page that actually needs wallet state renders); WalletButton is
    inside the same provider tree, so no extra plumbing needed.
- **Effort:** M
- **Status:** fixed (see commit)

### P2 — Polish / telemetry / DX

#### F-04 — Governance sync `written` metric always reads 0

- **Area:** sync telemetry
- **Evidence:** `backend/src/sync/governance-intake.ts:520-525`. The hot
  path increments `result.skipped++` on **both** the equal-skip branch
  AND the actual-PUT branch. The cold path (line 882) correctly
  increments `result.synced++`. Result: CloudWatch logs always show
  `written=0` even when DynamoDB shows ~2371 WCU/6h on `governance_actions`
  (~395 WCU/hr ≈ 30-80 Puts/hr). The Puts are legitimate (vote tallies
  shifting epoch-over-epoch), but operators can't observe the actual
  write rate from logs.
- **Fix:** Change the equal-vs-PUT branch on line 524 to
  `result.synced++` (the line-523 PUT is the same kind of write as the
  cold path).
- **Effort:** S
- **Status:** fixed (see commit)

#### F-05 — Hash-mismatch warnings now affect 4 actions, prompt referenced 1

- **Area:** data integrity
- **Evidence:** 4 actions across all statuses have `anchorVerified: false`
  — all 4 in `dropped` status: Pebble + Gerolamo, Cardano Treasury DeFi
  Liquidity Budget, Set 2025 Net Change Limit, HOSKY Hard Fork.
  The prompt's "1 known case (HOSKY)" reflects an older state.
  These are NOT new bugs — the surfacing logic (`anchorHashMismatch`
  flag added in v13) is working as intended; the count is just higher
  than the prompt noted.
- **Fix:** no code change. Captured here so the next audit-pass author
  doesn't chase a phantom regression.
- **Effort:** N/A
- **Status:** verified — no action

#### F-06 — IAM least-privilege violation on API Lambda role

- **Area:** security (defense-in-depth)
- **Evidence:** `infra/lib/api-stack.ts:95-106`. Every API Lambda
  shares `lambdaRole` which has `grantReadWriteData` on **every**
  DynamoDB table including `auth_nonces` (only auth handlers need
  it), `comments` (only comments handlers), `clubhouse_posts`
  (only clubhouse handlers), `audit_log` (only mutation handlers).
  A vulnerability in e.g. the `/governance` read handler would
  unnecessarily expose write access to user comments.
- **Fix:** split into per-domain roles or per-Lambda
  `grantReadWriteData`. Effort is the table-by-table mapping; not a
  one-line fix and not exploitable on its own.
- **Effort:** M
- **Status:** deferred — documented

#### F-07 — `/governance` response `total` field is misleading

- **Area:** API contract
- **Evidence:** `backend/src/handlers/governance/list.ts:64` returns
  `total: result.count` where `count` is the number of items returned
  in this page, not the total in the table. With `limit=20` and only
  20 active actions, it happens to coincide for `status=active`. For
  `enacted` (51 total) it would also lie. Frontend appears not to
  use this field, but a third-party client integration would mistake
  page-count for table-count.
- **Fix:** rename to `pageCount`, or replace with a true total (extra
  query) — keeping it cheap. Trivial doc fix at minimum.
- **Effort:** S
- **Status:** deferred — captured

#### F-08 — CSP allows `https://*.blockfrost.io` but frontend doesn't use Blockfrost

- **Area:** security hygiene
- **Evidence:** `infra/lib/frontend-stack.ts:91` — `connect-src 'self'
  https://api.drep.tools https://*.blockfrost.io`. `grep blockfrost
  frontend/src` returns zero matches; the deployed bundle contains zero
  `blockfrost` string occurrences. Phase B moved Blockfrost server-side
  only; the directive is dead weight that widens the CSP surface
  marginally.
- **Fix:** remove `https://*.blockfrost.io` from `connect-src`.
- **Effort:** S
- **Status:** fixed (see commit)

#### F-09 — TYPE label uppercased: "TREASURYWITHDRAWALS"

- **Area:** UX polish
- **Evidence:** `/governance/:id` shows `TREASURYWITHDRAWALS` as the
  action-type pill. The history page already maps to "Treasury
  Withdrawal" via `TYPE_LABELS` in `GovernanceHistoryPage.tsx:40-48`;
  the detail page just uppercases the raw `actionType` enum value.
- **Fix:** share the `TYPE_LABELS` map across detail + card components.
- **Effort:** S
- **Status:** deferred — captured (cosmetic, low-leverage)

#### F-10 — Governance-intake duration p95 = 28.8s, max = 36.9s

- **Area:** sync resilience
- **Evidence:** 200-sample distribution over 24h: min 905 ms, p50
  2689 ms, p95 28831 ms, max 36982 ms. 37/200 (18.5%) cycles ran >10s
  and 7/200 (3.5%) >30s. With a 10 min timeout there's no risk, but
  the gap between the warm path (~2s) and the long tail (~30s)
  suggests Koios occasionally responds slow. The 1-min cadence still
  works because of Lambda concurrent invocations; just worth tracking.
- **Fix:** no code change today; would benefit from adding a
  `Statistics: Max,P95` CloudWatch alarm to alert if either crosses
  60s (warning before timeout). Doc-only for now.
- **Effort:** S (alarm) / 0 (doc)
- **Status:** deferred — captured

## Verification methodology

For every finding below P3, I verified end-to-end before accepting:

- API contract claims: live `curl` against `https://api.drep.tools`.
- Vote-math identity: 10 actions across `active` + `enacted` × all 6
  action types — every single sample satisfies the documented
  `yes.power + no.power + notVoted.power == totalActive.power` identity
  with `abstain.power` separately reported.
- CSP correctness: live response header read from the deployed
  CloudFront distribution.
- Bundle sizes: read from the actual deployed `dist/`.
- Sync health: 24h log scan for `ERROR`, 200-cycle duration histogram,
  Blockfrost-call count, Koios 429 count (all clean).
- DynamoDB write rate: CloudWatch metrics per table over 6h.
- Frontend: Chrome MCP walk of /, /governance, /governance/:id,
  /governance/history, /dreps, /drep/:id, light/dark toggle, console.
